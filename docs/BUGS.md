# 缺陷记录

## BUG-20260717-01：窄屏下配对码撑破实例卡片

- 状态：已修复（v0.3.0）
- 优先级：P2
- 发现日期：2026-07-17
- 影响版本：`v0.2.1-dev.20260717.1.c447784.dirty`
- 影响范围：Web 控制台 → 运行实例 → 一次性配对码

### 现象

在约 440 px 宽的窄窗口中生成运行实例配对码后，配对码作为不可断行的长字符串超出卡片和页面内容区域，使整个页面出现横向滚动条。用户必须横向滚动才能查看页面，实例侧栏的移动端布局不可正常使用。

截图中包含一次性配对码，因此不将原图或完整配对码提交到仓库。

### 复现步骤

1. 打开 Web 控制台并进入“运行实例”。
2. 点击“添加实例”，生成一次性配对码。
3. 将浏览器视口缩小到 440 px 左右，或使用手机访问。
4. 观察配对码卡片和页面底部。

### 当前行为

- `#runtime-enrollment-code` 没有可断行或受限溢出样式。
- 配对码撑破 `.card` 和运行实例侧栏。
- 页面根级出现横向滚动条。

### 期望行为

- 320–480 px 宽度下，页面不得产生由配对码引起的根级横向滚动。
- 配对码应在卡片内部安全换行，或只在自身容器内部滚动。
- “复制”操作始终可见，并复制未经截断的完整配对码。
- 桌面布局和配对码的一次性使用语义保持不变。

### 验收标准

1. 在 320、375、440 和 480 px 视口生成配对码后，`document.documentElement.scrollWidth <= document.documentElement.clientWidth`。
2. 配对码所有可见内容都被限制在卡片边界内。
3. 点击“复制”所得内容与服务端返回的配对码完全一致。
4. 增加覆盖窄屏溢出约束的 Web 回归测试。

### 修复记录

- 配对码改为受限网格布局，并使用 `overflow-wrap:anywhere` 与 `word-break:break-all` 在卡片内换行。
- 配对卡片新增服务器一键接入命令，命令区自身换行且不会扩大页面宽度。
- Web 回归测试约束配对码不得恢复为无边界的内联布局。

## BUG-20260717-02：Linux 一键接入后 user service 以 218/CAPABILITIES 退出

- 状态：已修复（v0.3.0）
- 优先级：P1
- 发现日期：2026-07-17
- 影响版本：`v0.2.1-dev.20260717.2.c447784.dirty`
- 影响范围：Linux systemd user service、一键接入脚本

### 现象与原因

一键脚本可以创建专用用户、安装 Agent 并完成控制面配对，但 `submux-agent.service` 在创建本地 Socket 前退出，systemd 报告 `status=218/CAPABILITIES`。逐项临时单元测试确认 `ProtectKernelModules=true` 与 `ProtectKernelLogs=true` 不能由该无特权 user manager 执行；其余加固项组合正常。

安装器随后立即执行一次 `status`，还会把正常启动期间尚未创建 Socket 的短暂状态误报为失败。

### 修复与验收标准

1. user unit 不再包含 `ProtectKernelModules` 或 `ProtectKernelLogs`，但保留已验证可用的其他限制。
2. 配对完成后最多等待 20 秒，直到本地 Socket 和 `status` 可用。
3. `submux-agent.service` 保持 `active`，Socket 属于专用用户且不开放 TCP 管理端口。
4. 控制面实例状态变为在线，无需重新配对或重新生成设备身份。

## BUG-20260718-03：慢链路下载 Mihomo 被误报为资产大小不一致

- 状态：已修复（v0.3.0）
- 优先级：P1
- 发现日期：2026-07-18
- 影响版本：`v0.2.1-dev.20260717.3.c447784.dirty`
- 影响范围：rootless Agent → Mihomo 精确版本安装

### 现象与原因

在 aliyun 开发服务器安装官方 Mihomo `v1.19.28` 时，GitHub Release API 与下载响应均声明资产大小为 18,107,361 字节，但 Agent 在两分钟后返回“official asset size does not match release metadata”。实际原因是官方资产 HTTP 客户端的总超时只有两分钟，慢链路读取被中断；读取错误与真实大小不一致共用同一条错误信息。

### 修复与验收标准

1. 官方 Release 下载总超时调整为十分钟，仍保留 100 MiB 压缩资产上限与 250 MiB 解压后二进制上限。
2. 下载读取或超时错误单独返回，不再伪装成 Release 元数据不一致。
3. 继续严格验证固定官方仓库、精确 tag/资产名、受限重定向域名、GitHub SHA-256 digest 和实际字节数。
4. 在 aliyun 慢链路上可由普通用户完成 `v1.19.28` 安装并通过版本核对。

## BUG-20260720-04：暂停后恢复 Agent 日志会重复显示历史内容

- 状态：已修复（v0.3.0）
- 优先级：P2
- 发现日期：2026-07-20
- 影响版本：`v0.2.1-dev.20260720.4.c447784.dirty`
- 影响范围：Web 控制台 → 运行实例 → 日志 → Agent

### 现象与原因

暂停 Agent 实时日志后再次开始，Agent 会按设计先发送本地保留的日志尾部。页面没有记录已经处理的位置，把整个尾部再次追加到现有内容中，因此每恢复一次就重复显示一遍历史日志。

### 修复与验收标准

1. 同一 Agent 进程产生的日志带有稳定的流标识和单调递增的编号。
2. 页面分别保存 Agent 与 Mihomo 的日志内容及 Agent 日志读取位置。
3. 暂停后恢复、切换日志来源再切回时，只追加暂停期间产生的新 Agent 日志。
4. 清空显示后恢复日志不会重新显示已经清空的历史内容。
5. Agent 重启后使用新的流标识，新进程从编号 1 开始的日志仍可正常显示。
